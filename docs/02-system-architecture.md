# 02 - System Architecture

> Documento canonico de arquitectura. Define como se implementa el dominio en Next.js, Prisma, Supabase y Vercel sin agregar complejidad innecesaria.

---

## 1. Architecture Decision

La app sera un monolito fullstack con Next.js App Router.

No usamos monorepo, no usamos backend separado y no usamos paquetes internos hasta que exista una necesidad real.

```txt
Browser
-> Next.js App Router
-> Server Components / Server Actions / Route Handlers
-> Prisma
-> Supabase Postgres
```

Supabase se usa para:

- Anonymous Auth.
- Postgres hosted database.
- Dashboard operativo.
- Backups administrados.

Next.js se usa para:

- UI.
- Server-side reads.
- Server Actions.
- Admin endpoints.
- Cron endpoints.
- Scoring.
- API-Football sync.

Prisma se usa para:

- Schema.
- Migrations.
- Type-safe queries.
- Seed scripts.

## 3. Database Architecture

Supabase Postgres is the source of truth.

Prisma owns the app schema and migrations for public application tables.

Supabase Auth owns auth tables internally. The app does not create custom session tables.

### App tables

```txt
users
teams
players
matches
match_predictions
tournament_predictions
tournament_results
leaderboard_daily
api_sync_logs
```

### Removed tables

```txt
sessions
recovery_keys
phase_locks
groups
```

Reason:

- Sessions are owned by Supabase Auth.
- `recovery_key` is a field on `users`, not a separate table.
- Phase locking is derived from match/tournament dates and tournament progress.
- Groups are out of scope.

## 4. Prisma Strategy

`prisma/schema.prisma` is created only after `01-foundation-domain.md` and this file are stable.

Prisma models should map closely to the domain entities:

```txt
User
Team
Player
Match
MatchPrediction
TournamentPrediction
TournamentResult
LeaderboardDaily
ApiSyncLog
```

Important model decisions:

- `User.id` is not generated by Prisma during normal user creation; it receives the Supabase auth UID.
- Tournament prediction pick fields are nullable.
- `Match.winnerTeamId` is nullable.
- `MatchPrediction` has no `predictedWinnerId` in MVP.
- Points and exact hits are stored after scoring for fast reads.
- Leaderboard rows are snapshots, not live views.

## 5. Auth Architecture

### Strategy

Use Supabase Anonymous Auth only.

No email/password in MVP.
No custom session tokens.
No custom sessions table.

### Flow

```txt
User opens app
-> Supabase anonymous user is created/restored
-> Supabase stores/refreshes auth cookies
-> App checks if local users row exists
-> If missing, user goes through setup
-> Setup creates users row with id = auth.users.id
```

### Middleware

`src/middleware.ts` refreshes Supabase cookies and protects routes that require auth.

Middleware responsibilities:

- Keep Supabase session fresh.
- Let public/setup routes through.
- Redirect incomplete users to setup.
- Keep admin routes protected by admin logic, not user auth alone.

### Recovery key

`users.recovery_key` is generated on setup.

Recovery implementation can be refined later, but the architecture requirement is:

- user can present a recovery key
- server finds the matching local user
- app restores access flow through Supabase Auth

## 6. Data Access Strategy

### Reads

Default: Server Components fetch with Prisma.

Use this for:

- leaderboard initial render
- match lists
- current user
- tournament pick options

### Mutations

Default: Server Actions.

Use this for:

- create user profile
- submit match predictions
- submit tournament predictions
- update name/avatar

### System operations

Use Route Handlers under `app/api`.

Use this for:

- `/api/admin/*`
- `/api/cron/*`
- `/api/sync/*`
- `/api/scoring/*`

Route Handlers require `ADMIN_SECRET` or cron secret where appropriate.

## 8. Server Actions Plan

```txt
features/users/actions/
  create-user-profile.action.ts
  update-user-profile.action.ts

features/predictions/actions/
  submit-match-predictions.action.ts

features/tournament-predictions/actions/
  submit-tournament-predictions.action.ts
```

Server Actions must:

- read the authenticated Supabase user server-side
- validate input with Zod
- enforce ownership with `userId = session.user.id`
- enforce lock rules server-side
- use Prisma only on the server
- return small typed results

## 9. Locking Architecture

No cron is needed to lock predictions.

### Match lock

```ts
currentTime >= match.starts_at
```

This check is done:

- in UI for display only
- in Server Action before insert

Server check is the source of truth.

### Tournament lock

```txt
2026-06-11 00:00:00 America/Argentina/Buenos_Aires
```

This check is done in the tournament prediction Server Action.

---

## 10. Scoring Architecture

Scoring logic lives in `src/server/scoring`.

It should stay as pure functions where possible and be covered by Vitest.

Core functions:

```txt
calculateMatchScore()
calculateTournamentScore()
calculateLeaderboard()
```

When admin closes a match:

```txt
update match
-> calculate all match prediction points
-> update match_predictions
-> recalculate leaderboard_daily
```

Leaderboard recalculation can be full-table because the expected userbase is small.

## 11. External API Architecture

API-Football is an input source, not source of truth.

Use cases:

- initial teams seed
- initial players seed
- initial matches seed
- structural fixture sync

Not used for:

- live score
- automatic final scores
- official match closing
- realtime status
- player stats

Provider location:

```txt
src/server/football-api/
  football-api-provider.ts
  football-api-mappers.ts
  football-api-schemas.ts
```

Seed raw data location:

```txt
prisma/seed/raw-data/
  raw-teams.json
  raw-players.json
  raw-matches.json
```

## 12. Caching Architecture

Use TanStack Query for client-side server-state cache.

Use Zustand only for:

- local prediction drafts
- modal/UI state
- onboarding flags

Recommended query keys:

```txt
["matches", phase]
["leaderboard"]
["predictions", userId]
["tournament-prediction", userId]
```

Global client behavior:

- no polling by default
- no refetch on window focus
- retry: 1
- manual invalidation after submits/admin updates

---

## 13. Admin Architecture

Admin is intentionally simple.

No RBAC tables in MVP.

Admin operations are protected by `ADMIN_SECRET`.

Admin functions:

- close match manually
- recalculate leaderboard
- sync fixture structure
- inspect sync logs/system status
- soft reset test users/predictions during development

Admin is not part of the normal user navigation.

---

## 14. Cron Architecture

One Vercel Cron endpoint:

```txt
/api/cron/daily
```

Schedule:

```txt
03:00 America/Argentina/Buenos_Aires
```

Responsibilities:

- optional structural fixture sync
- daily leaderboard snapshot/recalculation
- write `api_sync_logs`

Cron must fail safely:

- no destructive writes
- no automatic score closing
- log errors
- allow manual admin retry

---

## 15. Validation And Errors

Zod is used at every input boundary:

- forms
- Server Actions
- Route Handlers
- external API payload mapping

Expected app errors:

- unauthenticated
- profile incomplete
- duplicate username
- prediction already submitted
- prediction locked
- match not found
- admin unauthorized
- external API failed

UI should show friendly errors, but server logs should keep enough context for debugging.

## 16. Environment Variables

```txt
DATABASE_URL
DIRECT_URL
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_ANON_KEY
API_FOOTBALL_KEY
ADMIN_SECRET
CRON_SECRET
```

Rules:

- browser can only receive `NEXT_PUBLIC_*`.
- Prisma uses `DATABASE_URL` and `DIRECT_URL`.
- API-Football key never reaches browser.
- Admin/Cron secrets never reach browser.

## 17. Deployment Architecture

Local first, then Supabase, then Vercel.

Order:

```txt
1. Finalize Foundation & Architecture
2. Create Prisma schema
3. Create Supabase project
4. Add env vars locally
5. Run Prisma migration
6. Confirm tables in Supabase
7. Configure Supabase Anonymous Auth
8. Build auth/user setup
9. Build predictions and leaderboard
10. Deploy to Vercel
11. Add env vars in Vercel
12. Configure Vercel Cron
```

Vercel is not needed until local DB/auth flow works.

---

## 18. Testing Architecture

Vitest for core business logic:

- scoring
- locking
- leaderboard sorting

Minimal E2E later with Playwright:

- create anonymous user/profile
- submit prediction
- view leaderboard

No real API-Football calls in tests.
No live production database in tests.

---

## 19. Architecture Priorities

1. Simple monolith.
2. Server-enforced prediction integrity.
3. Prisma schema matching the domain.
4. Supabase Auth only, no duplicate session system.
5. Manual admin control over official results.
6. Low API usage.
7. Fast local development.
